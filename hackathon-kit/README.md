# Hackathon Kit

Purpose: help PM-led teams build a real AI demo, not just a pitch deck.

The target output is:

```text
Repo + Demo + Eval + Decision Log + Backup Plan
```

## Principle

Problem first, but demo-able. A big idea that cannot produce evidence in 24-48 hours is not a good hackathon scope.

## PM Playbook

Use this kit to turn a hackathon from a pitch exercise into Builder training.

For an Agent-themed internal hackathon, PMs should translate the track into three questions:

1. What real workflow or product experience is broken today?
2. Why is an Agent necessary, not just decorative?
3. What can judges or users experience in 2-3 minutes?

Recommended PM workflow:

1. Start with [`problem-one-pager.md`](problem-one-pager.md): define one user, one scenario, one painful task, and one expected outcome.
2. Use [`demo-spine.md`](demo-spine.md): compress the idea into one visible path from input to result.
3. Use [`judging-rubric.md`](judging-rubric.md): map judging criteria to build priorities, especially value, Agent innovation, and hands-on experience.
4. Use [`eval-template.md`](eval-template.md): prepare 10-20 cases that prove the demo works beyond one lucky run.
5. Use [`decision-log.md`](decision-log.md): record what you cut, why, and what the demo does not prove yet.
6. Use [`demo-backup-plan.md`](demo-backup-plan.md): prepare for API, model, network, deployment, or hardware failure.

PMs should personally own at least one build artifact: Problem One-Pager, Demo Spine, Eval Cases, Decision Log, demo script, or backup plan. The goal is not to replace engineers. The goal is to become a Builder who can turn a vague opportunity into a real problem, runnable evidence, and a next decision.

## 百度内部 Hackathon 参赛经验

适合放在 PM Builder 转型之后讲：

- 不要从“我要用某个模型 / Agent 技术”开始，先从真实用户、业务流程或工作场景开始。
- 把 “Agent 重塑工作流” 翻译成一个可观察的流程前后对比。
- 把 “Agent 重构产品范式” 翻译成一个用户可以完成的新任务链。
- 第一小时先做历史作品和已有项目去重，再找差异化切入点。
- 2-3 分钟内必须看见问题、Agent 关键动作、可见结果和为什么更好。
- 提交前明确 Demo 证明了什么、没有证明什么、下一步真实验证是什么。
